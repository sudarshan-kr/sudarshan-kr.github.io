Projects

AWS Three Tier Web Architecture

In this architecture, a public-facing Application Load Balancer forwards client traffic to our web tier EC2 instances. The web tier is running Nginx webservers that are configured to serve a React.js website and redirects our API calls to the application tier’s internal facing load balancer. The internal facing load balancer then forwards that traffic to the application tier, which is written in Node.js. The application tier manipulates data in an Aurora MySQL multi-AZ database and returns it to our web tier. Load balancing, health checks and autoscaling groups are created at each layer to maintain the availability of this architecture.

EC2 Auto Scaling Group (ASG) Application Load Balancer (ALB) IAM S3 RDS VPC CloudWatch SNS Route 53 CloudFront

E-Kart Application Deployment on Kubernetes

I built a Java-based E-Kart application and deployed it on a private AWS EKS cluster using a complete end-to-end DevOps workflow, where I designed a secure AWS infrastructure with a custom VPC, private subnets, IAM roles, and a bastion host for controlled access. The application was packaged using Maven, containerized with Docker, and the image was pushed securely to AWS ECR, while all Kubernetes manifests were stored in GitHub with ArgoCD implementing a GitOps-based automated deployment process. External traffic was managed through the AWS ALB Ingress Controller for secure and reliable access, featuring Karpenter for intelligent autoscaling of nodes based on demand, with monitoring enabled via Prometheus and Grafana, and final access provided through a custom domain configured in AWS Route 53.

AWS GitHub Maven Jdk Docker ECR Kubernetes Karpenter Argo CD Prometheus Grafana Ingess

MEAN Stack CI/CD Deployment – Full-Stack CRUD Application

I built and deployed a full-stack MEAN (MongoDB, Express, Angular, Node.js) CRUD application using complete DevOps automation. The source code was maintained in GitHub and automatically pulled to an AWS EC2 Ubuntu instance through a Jenkins CI/CD pipeline. Both frontend and backend applications were containerized using Docker, while MongoDB was deployed using the official Docker image. Docker Compose was used to orchestrate and run all services together in a single deployment. Jenkins handled building images, pushing them to Docker Hub, and redeploying updated containers on the EC2 server. Nginx was configured as a reverse proxy to route frontend and backend traffic efficiently. Finally, the application was successfully deployed and made accessible via the EC2 public IP, ensuring a reliable and automated deployment workflow.
